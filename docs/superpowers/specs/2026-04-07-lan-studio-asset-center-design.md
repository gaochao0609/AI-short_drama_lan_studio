# LAN Studio 统一资产中心设计文档

日期：2026-04-07  
状态：已确认设计方向，待进入 spec review / implementation plan  
语言：中文  
适用仓库：`.worktrees/lan-studio-v1`

## 1. 背景与问题

当前 LAN Studio 已具备从脚本到分镜、图片、视频的基础工作流，但素材管理仍然是流程割裂式的：

1. 没有独立的项目资产中心，素材分散在项目详情、图片页、视频页和脚本结果中。
2. 分镜页目前只接受系统内的脚本版本，无法直接引用用户上传的剧本文件。
3. 图片页虽然支持项目内单张图片做图生图，但不支持先上传多张本地图片作为项目资产再统一引用。
4. 视频页支持从项目内已有图片多选参考图，但缺少统一的本地图片资产上传入口。
5. 系统生成结果与用户上传素材没有统一的资产抽象，后续扩展格式、引用关系和复用路径会越来越混乱。

本次设计的目标不是增加新的创作流程阶段，而是在现有 `Script -> Storyboard -> Images -> Videos` 工作流基础上，引入统一资产中心，让所有素材先进入项目资产层，再由各流程消费。

## 2. 设计目标

本次设计需要同时满足以下目标：

1. 建立统一的项目资产中心，集中管理用户上传素材和系统生成结果。
2. 让分镜、图片、视频三个流程都从同一套资产来源读取输入。
3. 允许流程页保留“快速改选”能力，但资产的存储、展示、分类、绑定都收敛在资产中心。
4. 为后续扩展剧本文件格式、图片参考策略、多素材引用和资产复用预留空间。
5. 保持当前工作流节奏和主要路由结构不被打散，避免一次性重做整站业务逻辑。

## 3. 用户已确认的设计决策

用户已明确确认以下约束：

1. 采用统一的“项目资产中心”方案，而不是每个流程页各自维护上传入口。
2. 资产中心需要同时覆盖以下四类对象：
   - 用户上传剧本文件
   - 第 1 步生成的剧本
   - 用户上传图片素材
   - 系统生成图片 / 视频结果
3. 各流程默认读取“当前已绑定资产”，但流程页内允许用户临时改选，不强制每次都回资产中心。
4. 项目资产中心是唯一素材来源和唯一真相源，流程页只负责消费资产并生成新产物。

## 4. 方案选择

### 4.1 备选方案

设计讨论中考虑过三种方向：

1. 重资产中心、弱流程页  
   资产中心承担全部上传、预览、设为输入操作，流程页只显示当前绑定素材。

2. 资产中心为主，流程页可局部改选  
   资产中心作为统一来源；流程页默认读取当前绑定资产，同时提供轻量选择器做局部改选。

3. 流程驱动中心  
   资产中心只做结果仓库，上传和引用逻辑仍然分散在各流程页。

### 4.2 最终选择

最终采用方案 2：`资产中心为主，流程页可局部改选`。

选择原因：

1. 能保证资产来源统一，避免继续扩散“每页一套上传逻辑”的问题。
2. 不会破坏创作连续性，用户仍可在流程页直接改选当前输入素材。
3. 能以较低风险接入现有页面和服务层，不需要一次性重写所有创作页面。
4. 最符合“统一资产中心”的产品目标，同时保留一线创作效率。

## 5. 范围与非目标

### 5.1 本次设计范围

本次设计覆盖以下内容：

1. 新增项目资产中心页面：`/projects/[projectId]/assets`
2. 为项目详情页增加资产概览和资产中心主入口
3. 为分镜、图片、视频页接入统一资产引用模型
4. 扩展资产数据模型，使其能描述来源、分类、元数据和流程绑定
5. 统一系统生成结果回流到资产中心

### 5.2 非目标

以下内容不在本次设计范围内：

1. 不新增新的创作阶段
2. 不重做脚本问答流程本身
3. 不在第一期支持复杂文档解析流水线、OCR 或 PDF 深度结构提取
4. 不把资产中心做成完整 DAM（Digital Asset Management）系统，例如批量标签体系、版本审批、跨项目共享库
5. 不改变现有用户权限模型，只在现有项目权限范围内做资产约束

## 6. 信息架构

### 6.1 项目详情页

项目详情页继续作为项目总控台，但新增一个 `资产概览` 区块，位于流程总览之后、历史记录之前。

资产概览至少展示：

1. 剧本资产数
2. 图片资产数
3. 视频资产数
4. 当前流程绑定概览
   - 分镜当前绑定剧本
   - 图片当前绑定参考图数量
   - 视频当前绑定参考图数量
5. 主入口按钮：`进入资产中心`

目的：

1. 让用户在项目入口页就看到“素材准备程度”
2. 强化资产中心是正式工作流的一部分，而不是附加功能

### 6.2 项目资产中心

建议新建路由：`/projects/[projectId]/assets`

页面结构：

1. 顶部摘要区
   - 项目名称
   - 资产总数与分类统计
   - 上传主按钮
   - 最近更新提示

2. 分类切换区
   - 全部
   - 用户上传剧本
   - 生成剧本
   - 用户上传图片
   - 生成图片
   - 生成视频

3. 资产列表区
   每张资产卡片至少包含：
   - 资产名称
   - 分类
   - 来源（上传 / 系统生成）
   - 时间
   - 预览 / 下载
   - 可用操作

4. 流程绑定操作
   资产卡根据类型显示不同按钮：
   - 设为分镜输入
   - 添加为图片参考
   - 添加为视频参考

资产中心承担以下职责：

1. 上传剧本文件
2. 上传图片素材
3. 展示系统生成剧本、图片、视频结果
4. 统一进行预览、下载、绑定、删除等管理动作

### 6.3 分镜页

分镜页不再只围绕 `scriptVersionId` 工作，而是接入统一“剧本资产”概念。

页面结构变化：

1. 输入来源区
   - 当前绑定剧本资产摘要
   - 快速改选入口
   - 如果用户上传的是剧本文件，则显示解析后的正文预览

2. 生成操作区
   - 生成分镜按钮
   - 当前任务状态

3. 结果区
   - 分镜段落
   - 视频提示词

分镜页允许引用两类剧本资产：

1. 第 1 步生成剧本
2. 用户上传剧本文件

### 6.4 图片页

图片页保留“文生图 / 图生图”模式，但参考素材统一来自资产中心。

需要支持：

1. 从用户上传图片资产中多选
2. 从系统生成图片中多选
3. 保留当前流程已绑定素材的默认展示
4. 在图片页内临时改选当前参考图

页面重心从“上传入口”转向“已绑定素材 + 当前提示词 + 结果列表”。

### 6.5 视频页

视频页与图片页保持同样的资产逻辑：

1. 当前绑定参考图来自资产中心
2. 支持多张图片作为视频参考
3. 页面内可临时改选，不重复维护上传入口
4. 视频结果自动回流资产中心

## 7. 数据模型设计

### 7.1 资产模型扩展

现有 `Asset` 模型可继续使用，但需要补充更明确的分类语义。

建议新增或规范以下字段语义：

1. `category`
   - `script_source`
   - `script_generated`
   - `image_source`
   - `image_generated`
   - `video_generated`

2. `origin`
   - `upload`
   - `system`

3. `metadata`
   用于存储扩展信息，例如：
   - 原始文件名
   - 文件扩展名
   - 解析状态
   - 提取出的剧本文本摘要
   - 尺寸、页数、引用次数

这套设计允许后续增加新的来源或资产类型，而不需要重做表结构。

### 7.2 流程绑定模型

建议新增项目级流程绑定配置，不与项目描述字段混放。

建议独立表或独立模型，例如 `ProjectWorkflowBinding`，至少包含：

1. `projectId`
2. `storyboardScriptAssetId`
3. `imageReferenceAssetIds`
4. `videoReferenceAssetIds`
5. `updatedAt`

原因：

1. 这是明确的流程状态，而不是普通项目元数据
2. 后续可能扩展更多绑定策略、默认模板和最近使用记录

### 7.3 剧本输入统一抽象

分镜服务层不再直接接受 `scriptVersionId`，而是统一接受：

1. `scriptAssetId`

服务层负责判断该资产属于哪种来源：

1. 系统生成剧本
2. 用户上传剧本文件

然后提取出最终可用于分镜生成的正文内容。

这是本次扩展的关键抽象点，必须优先落地。

## 8. 资产生命周期与回流规则

### 8.1 用户上传素材

用户上传后，资产进入资产中心并立即可见：

1. 上传剧本文件 -> 进入 `script_source`
2. 上传图片素材 -> 进入 `image_source`

### 8.2 系统生成结果

系统生成结果自动回流资产中心，不要求用户手动保存：

1. 第 1 步生成剧本 -> `script_generated`
2. 图片生成结果 -> `image_generated`
3. 视频生成结果 -> `video_generated`

### 8.3 流程消费

各流程消费资产的规则如下：

1. 分镜流程默认读取当前绑定剧本资产
2. 图片流程默认读取当前绑定图片参考资产集合
3. 视频流程默认读取当前绑定图片参考资产集合
4. 流程页内可临时改选，确认后更新绑定状态

## 9. 上传与解析规则

### 9.1 第一阶段支持的上传类型

剧本文件第一阶段建议支持：

1. `.txt`
2. `.md`

图片素材第一阶段建议支持：

1. 常见图片格式，如 `jpg / jpeg / png / webp`

### 9.2 暂缓项

以下内容建议放到后续阶段：

1. `.docx` 解析
2. `.pdf` 解析
3. OCR
4. 更复杂的素材批量导入

原因：

1. 当前第一期重点是统一资产中心的工作流闭环
2. 文档解析复杂度会显著提高依赖和异常处理成本

## 10. 页面交互规则

### 10.1 资产中心负责

1. 上传
2. 分类浏览
3. 预览
4. 下载
5. 删除
6. 设为流程输入

### 10.2 流程页负责

1. 展示当前绑定资产
2. 快速改选
3. 发起生成
4. 展示任务状态与结果

原则：

1. 管理在资产中心
2. 执行在流程页

## 11. 权限与校验

所有资产访问和流程绑定都必须受项目范围限制。

后端必须校验：

1. 所有被绑定的资产都属于当前项目
2. 当前用户有权访问该项目
3. 图片 / 视频参考资产类型符合当前流程要求
4. 分镜输入资产必须是可解析成文本的剧本资产

不能只依赖前端传来的资产 ID。

## 12. 分阶段实施建议

### Phase 1：资产中心底座

完成以下内容：

1. 扩展资产分类与来源模型
2. 新增流程绑定模型
3. 新建资产中心页面
4. 支持剧本文件上传、图片素材上传、系统结果展示

目标：

1. 所有资产先“进得来、看得见、能绑定”

### Phase 2：分镜流程接入资产中心

完成以下内容：

1. 分镜输入从 `scriptVersionId` 改为 `scriptAssetId`
2. 支持用户上传剧本文件作为分镜输入
3. 支持系统生成剧本作为分镜输入

目标：

1. 打通“剧本资产化”这条主干链路

### Phase 3：图片 / 视频流程接入资产中心

完成以下内容：

1. 图片页支持引用资产中心多张图片
2. 视频页支持引用资产中心多张图片
3. 所有结果自动回流资产中心

目标：

1. 让图片与视频流程完全建立在统一资产层之上

## 13. 验收标准

以下条件满足时，可视为本次设计达成目标：

1. 项目存在独立的资产中心页面
2. 用户上传剧本文件、图片素材都能进入统一资产中心
3. 第 1 步生成剧本与系统生成图片 / 视频结果都会自动沉淀到资产中心
4. 分镜流程可引用上传剧本文件和系统生成剧本
5. 图片流程可引用多张项目图片资产作为参考
6. 视频流程可引用多张项目图片资产作为参考
7. 项目详情页能显示资产概览和流程绑定摘要
8. 所有引用行为都经过项目范围校验
