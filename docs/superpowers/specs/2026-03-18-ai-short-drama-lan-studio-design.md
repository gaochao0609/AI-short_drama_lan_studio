# AI短剧创作局域网工作台 V1 设计文档

日期：2026-03-18  
状态：已确认，可进入实施规划  
语言：中文  
部署目标：Windows 10 单机局域网部署

## 1. 项目目标

构建一个面向 5-20 人局域网日常使用的 AI 短剧创作网站，部署在一台 Windows 10 主机上，局域网内用户通过浏览器访问。系统需要提供账号审批、登录鉴权、数据持久化落盘、文件资产管理和 4 条 AI 创作链路。

V1 的核心目标：

1. 提供管理员可控的账号创建、审批和登录能力。
2. 提供以“项目”为中心的创作工作台，而不是 4 个彼此割裂的工具页。
3. 打通以下 4 条异步生成链路：
   - 生成剧本
   - 剧本转 15 秒一段的 AI 视频分镜提示词
   - 图片生成与编辑
   - AI 视频生成
4. 所有文本、图片、视频、任务记录都需要持久化保存，服务重启后可恢复访问。
5. 所有模型接入都通过统一模型适配层，V1 初期接用户已有的本地中转或代理接口。

## 2. 已确认约束

### 2.1 使用场景

1. 使用人数：5-20 人局域网日常使用。
2. 部署方式：一台 Windows 10 主机部署，其他用户通过浏览器访问该主机。
3. 开发方式：使用 VS Code，采用 Codex 5.4 进行开发协作。

### 2.2 账号和权限

1. 不开放“提交后立即可用”的自助注册。
2. 允许提供注册申请页，但账号只有在管理员创建或审批后才能登录。
3. V1 角色仅保留两种：
   - `admin`
   - `user`

### 2.3 基础设施

1. 采用 Docker Desktop 运行核心服务。
2. 使用 `Postgres` 保存结构化业务数据。
3. 使用 `Redis` 作为任务队列和运行态缓存。
4. 使用本地文件目录保存上传文件、生成图片、生成视频和导出文件。

### 2.4 模型接入策略

1. 不直接把业务逻辑绑定到某个模型厂商。
2. 所有模型调用统一走模型适配层。
3. 初期通过管理员后台配置本地中转或代理接口。
4. 模型代理地址、密钥、默认模型、超时和重试参数均需要在管理员后台可配置。

## 3. V1 范围

### 3.1 V1 包含内容

1. 登录页、首次改密页、会话管理。
2. 注册申请页，提交后进入待审批状态，不自动开通。
3. 管理员后台：
   - 账号创建
   - 账号审批
   - 账号禁用
   - 密码重置
   - 模型配置管理
   - 任务监控
   - 存储管理
4. 创作工作区：
   - 仪表盘
   - 生成剧本页
   - 剧本转分镜页
   - 图片生成与编辑页
   - AI 视频生成页
   - 项目详情页
5. 项目、任务、文本版本、文件资产的持久化。
6. 异步任务执行、任务状态跟踪、失败记录和重新入队。

### 3.2 V1 不包含内容

1. 自助注册。
2. 提交后无需管理员审批即可直接登录。
3. 短信或邮箱验证码。
4. 多组织、多团队、细粒度权限体系。
5. 在线多人协作编辑。
6. 自动把多个 15 秒视频拼接成成片。
7. 计费、套餐、积分、充值等商业化系统。
8. 云对象存储，V1 只用本地磁盘。

## 4. 推荐总体方案

采用“模块化单体 + 内置异步队列 + 本地文件存储”的方案。

### 4.1 方案选择结论

选用方案：`单体全栈 + 内置任务队列 + 本地存储`

推荐理由：

1. 对 `Windows 10 单机 + 5-20 人局域网` 最稳妥。
2. 前后端、数据、任务、文件管理可以放在一个代码库中统一推进。
3. 图片和视频生成可以通过异步队列执行，避免阻塞 Web 请求。
4. 后续如果人数增加，可逐步把 Worker 或模型网关单独拆分，而不需要推翻 V1 架构。

### 4.2 技术栈建议

1. Web 框架：`Next.js`
2. ORM：`Prisma`
3. 数据库：`Postgres`
4. 队列：`BullMQ + Redis`
5. 认证方式：本地账号 + 数据库有状态会话 + `HttpOnly Cookie`
6. 文件存储：宿主机本地目录挂载
7. 部署方式：`Docker Compose` 管理 `web + postgres + redis`

## 5. 系统边界与架构

系统由 6 个核心层组成：

1. `Web 应用层`
   - 负责页面渲染、表单提交、任务进度展示、文件预览和下载
2. `业务 API 层`
   - 负责认证鉴权、项目管理、任务创建、任务查询、模型配置管理
3. `任务编排层`
   - 负责把用户操作转成标准异步任务，并调度对应 Worker
4. `模型适配层`
   - 负责统一不同模型的输入输出格式，对接本地代理接口
5. `持久化层`
   - Postgres 保存结构化数据
   - Redis 保存队列和短期运行态
   - 本地磁盘保存文件资产
6. `管理员控制层`
   - 负责账号治理、模型治理、任务治理和存储治理

## 6. 页面与功能设计

### 6.1 认证区

1. 登录页
   - 用户名或邮箱
   - 密码
   - 登录失败提示
2. 注册申请页
   - 用户填写基础信息并提交申请
   - 页面只提示“等待管理员审批”
   - 不提供自动激活能力
3. 首次改密页
   - 管理员创建的初始密码首次登录后必须修改
4. 会话过期处理
   - 会话过期后自动跳转登录页

### 6.2 创作工作区

1. 仪表盘
   - 最近项目
   - 最近任务
   - 失败任务提示
   - 磁盘占用概览

2. 生成剧本页
   - 输入：用户的模糊短剧创意
   - 执行方式：调用 `Claude Opus 4.6`
   - 方法：采用苏格拉底式多轮提问，引导用户逐步明确人物、冲突、结构、情绪走向、结局等要素
   - 流程类型：这是“会话型工作流”，不是一次性后台长任务
   - 交互保护：需要提供“重新生成当前问题”按钮，仅重试当前轮次，不引入复杂树状分支
   - 结果：在用户完成多轮回答后生成完整短剧剧本文本，并保存澄清问答记录

3. 剧本转分镜页
   - 输入：完整剧本
   - 执行方式：调用 `Gemini 3.1 Pro Preview`
   - 结果：输出 15 秒一段的分镜提示词列表
   - 每段至少包含：
     - 段落序号
     - 时长
     - 场景说明
     - 镜头说明
     - 人物动作
     - 台词或旁白
     - 视频生成提示词

4. 图片生成与编辑页
   - 输入模式：
     - 纯提示词文生图
     - 图片 + 提示词图生图
   - 执行方式：调用 `nano banana pro`
   - 结果：生成图片并保存输入参数、输入图、输出图之间的关联关系

5. AI 视频生成页
   - 输入：
     - 视频分镜提示词
     - 参考图片
   - 执行方式：调用 `seedance 2.0`
   - 结果：生成单条视频片段并保存文件路径、参数和任务日志

6. 项目详情页
   - 集中展示同一项目下的：
     - 剧本版本
     - 分镜版本
     - 图片素材
     - 视频素材
     - 任务历史
     - 下载入口

### 6.3 管理员区

1. 账号管理
   - 创建账号
   - 审批注册申请
   - 审批账号
   - 禁用账号
   - 重置密码
   - 强制首次改密

2. 模型配置
   - 配置代理接口地址
   - 配置 API Key 或鉴权头
   - 配置默认模型名
   - 配置启用状态、超时和重试次数
   - 为每条业务链路指定默认模型

3. 任务监控
   - 查看排队中、运行中、失败、成功任务
   - 查看失败原因
   - 查看重试记录
   - 手动重新入队
   - 手动取消任务

4. 存储管理
   - 查看磁盘目录占用
   - 查看项目文件体积
   - 清理失效临时文件
   - 查看磁盘剩余空间
   - 一键清理 30 天前的旧任务缓存图和中间产物

## 7. 核心业务模型

系统以“项目”为中心，不将 4 个页面设计成孤立工具。

### 7.1 项目

`projects` 是创作容器。一个项目下可以挂多个剧本版本、分镜版本、图片资产、视频资产和任务记录。

### 7.2 任务

每次用户点击“生成”都会创建一个异步任务。所有任务统一状态：

1. `queued`
2. `running`
3. `succeeded`
4. `failed`
5. `canceled`

### 7.3 产物

1. 文本产物
   - 剧本初始想法
   - 澄清问答记录
   - 最终剧本
   - 分镜结果
2. 文件产物
   - 上传原图
   - 生成图片
   - 生成视频
   - 导出包

## 8. 数据库实体设计

建议最少包含以下实体：

1. `users`
   - 用户基础信息
   - 密码哈希
   - 角色
   - 状态
   - 是否需要首次改密

2. `account_requests`
   - 注册申请信息
   - 申请状态
   - 审批人
   - 审批时间

3. `sessions`
   - 会话令牌
   - 用户 ID
   - 过期时间
   - 最后活跃时间
   - 来源 IP
   - 是否已失效

4. `projects`
   - 项目名称
   - 项目描述
   - 所属用户
   - 创建时间
   - 更新时间

5. `script_sessions`
   - 所属项目
   - 发起用户
   - 当前会话状态
   - 已完成轮次
   - 当前问题
   - 问答记录 JSON
   - 最终剧本版本引用

6. `script_versions`
   - 所属项目
   - 原始创意输入
   - 澄清问答 JSON
   - 最终剧本正文
   - 使用模型信息

7. `storyboard_versions`
   - 所属项目
   - 关联剧本版本
   - 分镜结构化内容 JSON
   - 使用模型信息

8. `assets`
   - 所属项目
   - 关联任务
   - 文件类型
   - 原始文件名
   - 存储路径
   - MIME 类型
   - 文件大小
   - 元信息 JSON

9. `tasks`
   - 所属项目
   - 发起用户
   - 任务类型
   - 任务状态
   - 输入摘要
   - 输出摘要
   - 错误摘要
   - 开始时间
   - 结束时间

10. `task_steps`
   - 关联任务
   - 步骤名
   - 重试次数
   - 状态
   - 日志
   - 原始响应摘要

11. `model_providers`
   - 提供方标识
   - 代理地址
   - API Key 或密钥引用
   - 默认模型名
   - 超时
   - 重试次数
   - 是否启用

## 9. 4 条链路的数据落盘要求

### 9.1 剧本生成

必须保存：

1. 用户原始模糊想法
2. 每一轮苏格拉底式问题
3. 每一轮用户回答
4. 最终剧本正文
5. 使用的模型、参数和任务日志
6. 当前会话轮次与是否已完成

### 9.2 剧本转分镜

必须保存：

1. 输入剧本版本引用
2. 每段分镜的结构化字段
3. 最终可供视频生成使用的提示词
4. 使用的模型、参数和任务日志

### 9.3 图片生成与编辑

必须保存：

1. 文生图或图生图模式
2. 输入提示词
3. 输入参考图
4. 输出图片
5. 生成参数
6. 输入输出文件关联关系

### 9.4 AI 视频生成

必须保存：

1. 输入分镜提示词
2. 输入参考图
3. 输出视频
4. 生成参数
5. 任务日志

## 10. 模型适配层设计

模型适配层的目标是让业务代码不关心底层模型的请求格式。

### 10.1 统一请求结构

建议统一为以下字段：

1. `taskType`
   - `script_question_generate`
   - `script_finalize`
   - `storyboard_split`
   - `image_generate`
   - `image_edit`
   - `video_generate`
2. `providerKey`
3. `model`
4. `inputText`
5. `inputFiles`
6. `options`
7. `traceId`

### 10.2 统一响应结构

建议统一为以下字段：

1. `status`
2. `textOutput`
3. `fileOutputs`
4. `usage`
5. `rawResponse`
6. `errorCode`
7. `errorMessage`

### 10.3 配置原则

1. 每条业务链路都可以配置默认模型。
2. 所有配置在管理员后台维护。
3. 模型代理接口的地址、鉴权、超时和重试参数都可调整。
4. 业务层只依赖统一适配接口，不直接处理不同厂商差异。

## 11. 队列与并发设计

V1 需要用异步队列隔离长耗时任务，尤其是图片和视频生成。

说明：`生成剧本` 采用“前台会话 + 后台小步调用”的模式，不把整段苏格拉底式澄清过程粗暴塞进一次长任务。

### 11.1 队列划分

建议拆成 4 条队列：

1. `script-queue`
2. `storyboard-queue`
3. `image-queue`
4. `video-queue`

### 11.2 设计原因

1. 文本任务可以设置更高并发。
2. 图片和视频任务需要更低并发，避免单机资源被拖满。
3. 失败重试策略可按任务类型单独控制。

### 11.3 任务执行原则

1. 用户提交请求后立即创建任务记录。
2. 剧本生成页的单轮提问与最终定稿可通过短任务驱动，但整个剧本澄清过程由会话状态控制。
3. 除剧本会话外，其余长耗时生成请求由 Web 负责入队，不等待模型生成完成。
4. Worker 消费队列并更新任务状态。
5. V1 明确采用短轮询查询任务进度，不使用 SSE。
6. 建议前端每 3 秒请求一次 `/api/tasks/[taskId]`，在 5-20 人局域网规模下优先保证实现稳定性。

## 12. 失败恢复策略

### 12.1 自动重试

1. 文本类任务自动重试 2 次。
2. 图片和视频类任务自动重试 1 次。

### 12.2 状态更新原则

1. 每一步先落库。
2. 再更新任务状态。
3. 最后通知前端。

### 12.3 文件写入原则

1. 先写入临时目录。
2. 校验成功后移动到正式目录。
3. 失败时清理临时文件并记录错误日志。

### 12.4 管理员恢复手段

1. 失败任务支持查看错误原因。
2. 失败任务支持手动重新入队。
3. 配置错误修复后，可在后台重新执行失败任务。

### 12.5 错误分类

以下错误视为可重试：

1. 代理接口超时
2. 代理接口返回 5xx
3. 网络瞬时失败

以下错误直接失败：

1. 参数不合法
2. 鉴权错误
3. 管理员配置缺失

## 13. 持久化与文件目录设计

### 13.1 数据持久化分层

1. `Postgres`
   - 用户、会话、项目、任务、文本版本、模型配置、文件元数据
2. `Redis`
   - 队列、任务进度缓存、限流和并发控制
3. `本地磁盘`
   - 上传文件、生成图片、生成视频、导出文件

### 13.2 文件目录建议

建议按项目和任务分层，例如：

1. `storage/uploads/{projectId}/{taskId}/`
2. `storage/generated-images/{projectId}/{taskId}/`
3. `storage/generated-videos/{projectId}/{taskId}/`
4. `storage/exports/{projectId}/`

这样便于：

1. 排查问题
2. 归档导出
3. 清理过期文件

## 14. 安全与权限基线

V1 不做复杂安全体系，但必须有基本边界：

1. 普通用户不能访问管理员页面。
2. 普通用户不能修改模型配置。
3. 普通用户不能查看其他用户项目。
4. 密码必须以哈希形式保存。
5. 会话必须以数据库有状态会话为准，并通过 `HttpOnly Cookie` 传递会话标识。
6. 上传文件需要做类型和大小校验。
7. 文件下载必须校验访问权限。

## 15. 部署与运维设计

### 15.1 部署方式

采用 `Docker Compose` 启动：

1. `web`
2. `worker`
3. `postgres`
4. `redis`

持久化要求：

1. `Postgres` 数据目录必须挂载 volume。
2. `Redis` 必须开启持久化并挂载 volume。
3. 本地文件目录必须通过宿主机挂载，避免容器重建后丢失。
4. 对于 Windows 10 + Docker Desktop，强烈建议将项目放在 WSL2 的 Linux 文件系统中运行，例如 `\\wsl$\\Ubuntu\\home\\<user>\\ai-studio`，再通过 VS Code Remote WSL 开发和启动容器，避免 NTFS bind mount 带来的 I/O 抖动和 `EPERM` 权限问题。

### 15.2 局域网访问方式

1. Web 服务监听固定端口。
2. Windows 主机开放该端口供局域网访问。
3. 用户通过浏览器访问该主机 IP 和端口。

### 15.3 启动时序与稳定性

1. 仅使用 `depends_on` 不足以保证数据库完全就绪。
2. `web` 和 `worker` 启动前需要等待 `Postgres` 和 `Redis` 进入可连接状态。
3. 应在容器编排或应用初始化中加入健康检查和连接重试机制，避免冷启动时 Prisma 连库失败。

### 15.4 备份建议

1. `Postgres` 定期导出。
2. `storage` 目录定期备份。

### 15.5 日志建议

日志至少分为：

1. 应用日志
2. 任务日志
3. 模型调用日志

管理员后台需要能查看最近失败任务的关键信息。

### 15.6 大文件与媒体预览约束

1. 资产下载接口必须支持视频文件的 HTTP Range 请求，确保局域网内 `<video>` 预览可拖动、可快进。
2. 图片上传接口必须定义明确的单文件大小上限，并在前后端同时校验。
3. V1 可接受对参考图上传采用受控大小上限的表单上传，但必须避免无上限地将大文件整体读入内存。

## 16. 测试与验收标准

### 16.1 功能验收

1. 登录、审批、禁用、密码重置可用。
2. 4 条业务链路都可以创建任务。
3. 任务状态能正确从 `queued` 进入 `running`，最终进入 `succeeded` 或 `failed`。
4. 成功结果能在项目详情页重新访问。

### 16.2 持久化验收

1. 服务重启后，用户、项目、任务、文本、图片、视频都仍可访问。
2. 文件路径与数据库元数据一致。

### 16.3 并发验收

1. 在 5-10 个并发任务下，系统不应出现整体卡死。
2. 图片和视频任务不应阻塞登录、浏览、项目查看等基础操作。

### 16.4 失败恢复验收

1. 代理接口不可用时，任务应进入 `failed` 并保留错误原因。
2. 修复模型配置后，管理员可重新入队失败任务。

### 16.5 权限验收

1. 普通用户无法进入管理员页面。
2. 普通用户无法修改模型配置。
3. 普通用户无法访问其他用户的项目数据。

## 17. 实施优先级建议

实施时建议按以下顺序推进：

1. 项目骨架、部署底座、数据库和认证
2. 管理员后台与账号治理
3. 项目、任务、文件资产基础能力
4. 模型适配层与模型配置管理
5. 剧本生成链路
6. 剧本转分镜链路
7. 图片生成与编辑链路
8. AI 视频生成链路
9. 任务监控、失败恢复和存储治理

## 18. 结论

V1 的正确定位不是“4 个独立按钮页面”，而是“一个以项目为中心、可持久化、可恢复、可管理的 AI 短剧创作局域网工作台”。

采用 `Next.js + Postgres + Redis + BullMQ + 本地文件存储 + Docker Compose` 的模块化单体方案，可以在当前约束下平衡开发效率、部署复杂度和后续扩展空间，并满足：

1. Windows 10 单机局域网部署
2. 5-20 人日常使用
3. 管理员审批账号
4. 4 条完整 AI 创作链路
5. 数据与文件持久化
6. 统一模型适配层与后台配置

下一步应基于本文档输出详细实施计划，将系统拆成可执行任务、验证步骤和边界明确的开发阶段。
