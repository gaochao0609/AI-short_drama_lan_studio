# LAN Studio UI 改版设计文档

日期：2026-04-03  
状态：已确认，待进入 implementation plan  
语言：中文  
适用仓库：`.worktrees/lan-studio-v1`

## 1. 背景与目标

LAN Studio 当前前端已经具备基础工作流，但视觉语言仍停留在“原型可用”阶段，主要问题包括：

1. 全局设计 token 很少，页面普遍依赖内联样式，缺乏可复用的视觉系统。
2. `workspace` 首页、项目详情页、登录页、后台页之间缺少统一品牌感。
3. 现有界面更像普通后台，而不是面向 AI 短剧创作的小型工作室。
4. 多处中文文案存在乱码，直接破坏使用体验。
5. 信息呈现仍偏“字段罗列”，不足以突出当前项目、流程进度、素材结果和下一步动作。

本次改版的目标不是重写业务流程，而是在保留现有产品结构和数据能力的前提下，把前端整体提升为“创作型工作室界面”，具体目标如下：

1. 建立一套覆盖工作区、项目页、登录页、后台页的统一设计系统。
2. 把 `workspace` 首页改造成“今日创作控制台”，突出当前项目、流程推进和快捷创建。
3. 把项目详情页改造成“制作台 / 流程入口页”，弱化纯历史列表，强化 `Script -> Storyboard -> Images -> Video` 四段工作流。
4. 统一登录页与后台页风格，使其与工作区属于同一产品，但保留各自信息密度差异。
5. 修复关键页面中的乱码文案，确保中文界面可读。

## 2. 已确认设计方向

用户已在 2026-04-03 的设计讨论中确认以下结论：

1. 整体风格选择“创作型”，而不是纯控台型或纯管理后台型。
2. 在三套候选视觉方向中，最终选择 `A. Spotlight Studio` 作为整站主方向。
3. 改版优先级为：
   - `workspace` 首页
   - 项目详情 / 流程入口页
   - 登录页与后台页统一风格

`Spotlight Studio` 的核心特征：

1. 深色舞台式背景，强调“创作焦点”而不是“纯数据罗列”。
2. 用聚光金和冷紫蓝做层次和重点，而不是继续沿用偏土色的旧方案。
3. 强调项目卡、素材预览、流程轨道、状态标签的戏剧感和可扫描性。
4. 登录页和品牌入口页具备明显气质，后台页则在同一视觉系统下做信息收敛。

## 3. 范围与非目标

### 3.1 本次改版范围

本次改版覆盖现有前端代码中的以下界面层：

1. 全局设计 token 与全局样式入口
2. 根布局、工作区布局、后台布局
3. `workspace` 首页与项目创建入口
4. 项目详情 / 流程入口页
5. 登录页
6. 注册申请页、首次改密页的风格统一
7. 后台页的壳层与主要管理页风格统一：
   - 用户管理
   - Provider 配置
   - 任务监控
   - 存储管理
8. 工作流子页面的视觉统一：
   - script
   - storyboard
   - images
   - videos

### 3.2 本次改版非目标

以下内容不在本次设计范围内：

1. 不改变现有业务路由、接口契约和数据库结构。
2. 不新增全新的创作流程或后台模块。
3. 不重做脚本、分镜、图片、视频的业务逻辑，只调整壳层、布局、组件与样式表达。
4. 不引入额外的前端 UI 框架或重型设计系统依赖。
5. 不把项目详情页扩展为完整素材编辑器或时间线系统。

## 4. 设计原则

本次改版遵循以下原则：

1. 保留现有产品模型，优先重构信息表达，而不是重构功能定义。
2. 让用户一眼看出“当前创作焦点是什么、下一步该做什么”。
3. 在创作气质和后台效率之间做平衡：工作区和项目页更有戏剧感，后台页更克制。
4. 优先使用统一 token 和复用组件，减少内联样式分散定义。
5. 在深色主题下单独校验可读性、焦点样式、状态对比度和移动端点击区。

## 5. 信息架构与页面优先级

### 5.1 `workspace` 首页

`workspace` 首页是本次改版的第一优先级，目标是从“基础列表页”升级为“今日创作控制台”。

页面结构：

1. 顶部 `spotlight hero`
   - 显示欢迎语、最近活动摘要、当前主要项目或最近更新项目
   - 提供核心 CTA：创建项目
   - 同时展示四段工作流总览：`Script / Storyboard / Images / Video`
2. 项目区
   - 用项目卡代替简单文本列表
   - 每张卡包含项目标题、状态、更新时间、当前阶段、下一步动作
3. 任务区
   - 展示最近任务、失败任务和关键状态
   - 强调扫描效率，适合快速发现卡住的任务
4. 创建项目区
   - 作为首页重要入口，而不是普通边缘表单卡
   - 清晰说明创建项目后会进入脚本流程

### 5.2 项目详情 / 流程入口页

项目详情页是本次改版的第二优先级，也是最重要的创作页。

页面结构：

1. 项目头部
   - 项目标题
   - 创意简介
   - 项目状态
   - 最近更新时间
   - 主 CTA 与次 CTA
2. 四段制作轨道
   - `Script`
   - `Storyboard`
   - `Images`
   - `Video`
   每段都应展示：
   - 当前阶段说明
   - 最近版本或最近产物摘要
   - 数量 / 状态
   - 进入对应工作流的操作入口
3. 历史内容区
   - 脚本版本
   - 分镜版本
   - 图片素材
   - 视频素材
   - 任务历史

布局原则：

1. 先展示“如何继续推进项目”，再展示“历史记录”。
2. 素材预览卡需要足够视觉权重，不能退化成后台附件列表。
3. 任务历史保留完整性，但不应抢占页面主视觉。

### 5.3 登录页与认证页

登录页是本次改版的第三优先级，但仍需明显纳入同一品牌语言。

要求：

1. 保留单卡片登录结构，避免过度复杂。
2. 通过背景、字体、按钮、品牌文案与工作区形成统一气质。
3. 注册申请页与首次改密页复用同一认证壳层与表单组件。
4. 修复所有乱码文案。

### 5.4 后台页

后台页与工作区共享视觉系统，但要比创作页更克制。

要求：

1. 保留侧边导航模式。
2. 统一页面头部、表格/列表容器、警告条、状态标签、按钮体系。
3. 延续深色舞台体系的主色逻辑，但后台正文区域需要更强可读性和更稳定的密度控制。
4. 不能为了追求创作感损失管理页的扫描效率。

### 5.5 工作流子页面

`script`、`storyboard`、`images`、`videos` 页面不做新的业务拆分，但应纳入同一套壳层与设计 token，确保从入口页进入后视觉连续。

要求：

1. 顶部标题与上下文信息统一。
2. 卡片、输入区、结果区、空状态、错误状态统一样式。
3. 工作流之间保持明显共性，但不抹平各自内容结构。

## 6. 视觉系统

### 6.1 色彩系统

主方向为“深色舞台 + 聚光高亮”。

建议 token：

1. `--bg`: 全局主背景，接近深蓝黑
2. `--bg-elevated`: 更高一级背景层，用于局部区块
3. `--panel`: 默认卡片背景
4. `--panel-strong`: 强调卡片背景
5. `--text`: 主文本
6. `--text-muted`: 次级文本
7. `--border`: 边框和分隔线
8. `--accent-gold`: 聚光金，用于主 CTA、焦点和高优先级标签
9. `--accent-violet`: 冷紫蓝，用于辅助高亮和流程层级
10. `--success`
11. `--warning`
12. `--danger`

色彩原则：

1. 不继续沿用旧版米棕色作为主品牌色。
2. 状态不能只用颜色表达，必须同时提供文字和层级差异。
3. 深色主题下需逐项校验文字对比度。

### 6.2 字体系统

字体选择参考 `ui-ux-pro-max` 的 `design-system` 结果：

1. 正文字体：`Fira Sans`
2. 功能性字体：`Fira Code`

使用规范：

1. `Fira Sans` 用于正文、标题、按钮、说明文案。
2. `Fira Code` 仅用于任务 ID、步骤标签、数字指标、时间戳、短状态标签。
3. 正文最小 16px，避免移动端缩放问题。
4. 保持统一字体层级，避免现有页面中的随意字号。

### 6.3 表面、边框与层级

1. 建立统一圆角等级，例如：
   - 页面大卡：24px
   - 常规卡片：20px
   - 内层内容卡：16px
   - 按钮 / 输入：14px 或胶囊形
2. 使用统一阴影和发光层级。
3. 重点区域允许使用轻微渐变和发光，普通卡片必须收敛。
4. 避免整站每块都强强调，重点只留给 hero、当前项目、主 CTA、关键流程入口。

### 6.4 动效系统

动效只服务于理解，不作为装饰堆叠。

允许的动效：

1. 页面首屏轻微淡入
2. 卡片 hover / focus 上浮
3. 按钮 hover / press 反馈
4. 流程状态切换时的过渡

约束：

1. 时长控制在 150ms 到 250ms
2. 尊重 `prefers-reduced-motion`
3. 不用大幅位移动画和无意义闪烁

## 7. 共享组件规范

本次改版需要抽出并统一以下组件模式：

1. 页面 shell
2. 侧边导航
3. 页面标题区 / hero / spotlight
4. 状态 badge
5. 指标卡
6. 项目卡
7. 流程轨道卡
8. 列表容器 / 结果栈
9. 空状态
10. 表单输入
11. 主按钮 / 次按钮 / 危险按钮
12. 提醒条 / 错误提示 / 成功提示

组件要求：

1. 样式来源应统一，而不是每页自带一套内联样式。
2. 组件 API 要贴近当前代码结构，避免为了视觉重构而制造不必要抽象。
3. 后续所有页面都应尽量通过共享类名或小型复用组件继承视觉系统。

## 8. 响应式与可访问性规则

### 8.1 响应式

1. 桌面端保留左侧导航。
2. 平板端压缩边距和栏宽。
3. 手机端切为单列内容，导航改为顶部品牌栏或紧凑头部。
4. 流程轨道在手机端纵向堆叠。
5. 所有主操作按钮点击区至少 44px 高。
6. 素材预览区在小屏上优先保留内容宽度，不出现横向滚动。

### 8.2 可访问性

1. 深色主题下文字与背景对比度满足 WCAG AA。
2. 键盘焦点样式在深色背景上必须清晰可见。
3. 表单错误贴近字段展示，并保留语义化错误提示。
4. 标题层级连续，不跳级。
5. 状态表达不能只依赖颜色。
6. 图片与视频预览必须保留合理的替代文本或标签。

## 9. 实现约束与迁移要求

### 9.1 技术约束

1. 继续使用当前 `Next.js App Router` 结构。
2. 不引入新的前端组件库。
3. 优先通过 `src/app/globals.css` 建立全局 token 和基础样式层。
4. 逐步减少页面中的大段内联样式。
5. 保持现有接口调用方式和服务层不变。

### 9.2 内容与代码迁移

1. 修复关键页面中的乱码中文文案：
   - `workspace`
   - 项目详情页
   - 登录页
   - 后台导航与提示文案
2. 根布局、工作区布局、后台布局需要统一到同一视觉系统。
3. 页面优先顺序必须与用户确认的优先级一致。
4. 改版过程中不能破坏现有表单提交、跳转、鉴权和列表渲染行为。

## 10. 验收标准

只有当以下条件满足时，这次 UI 改版才可视为达到设计目标：

1. `workspace` 首页已从简单仪表盘改为具备创作焦点、流程总览和项目卡的创作控制台。
2. 项目详情页已从历史列表页改为四段工作流入口明确的制作台。
3. 登录页、认证页、后台页与工作区共享统一设计语言。
4. 全局样式 token 已建立，主要页面不再各自复制色值和卡片样式。
5. 关键中文文案不再乱码。
6. 页面在桌面和移动端均可正常使用，不出现明显布局断裂或横向溢出。
7. 深色主题下的文本、焦点和状态具备足够可读性。

## 11. 规划准备结论

该设计已经具备进入 implementation plan 的条件。

规划阶段需要按以下顺序拆分：

1. 建立全局 token、字体、shell 和共享组件基础
2. 改造 `workspace` 首页与创建项目入口
3. 改造项目详情 / 流程入口页
4. 改造登录页、注册申请页、首次改密页
5. 统一后台页壳层与管理页风格
6. 将工作流子页面纳入同一视觉系统

以上拆分可直接作为下一步 implementation plan 的输入。
